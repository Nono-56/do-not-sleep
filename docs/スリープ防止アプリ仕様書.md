# Windows向けスリープ防止アプリ仕様書

## 1. 概要

### 1.1 目的
Windows PCが自動的にスリープ状態へ移行することを防ぐため、一定間隔で擬似的な入力を行う軽量なPythonアプリを開発する。  
ユーザーはGUIから動作方式、実行間隔、終了時刻などを設定でき、最終的には`.exe`形式で単体起動できることを目標とする。

### 1.2 想定利用シーン
- 長時間の監視作業中にPCをスリープさせたくない場合
- プレゼンテーションや表示端末の待機状態を避けたい場合
- 業務アプリの常時表示中に最小限の擬似入力を行いたい場合

### 1.3 開発方針
- Pythonで実装する
- Windows環境を対象とする
- GUIで主要設定を完結できるようにする
- 軽量性を重視し、常駐時のメモリ使用量は数MB台を目標とする
- `.exe`化してPython未導入環境でも起動可能にする

## 2. 対象環境

### 2.1 対応OS
- Windows 10
- Windows 11

### 2.2 実行形態
- 開発時: Pythonスクリプトとして実行
- 配布時: `.exe`ファイルとして実行

## 3. 機能要件

### 3.1 擬似入力機能
アプリは指定間隔ごとに、以下いずれかの動作を行う。

- キー入力
- 微小なマウス移動

#### 3.1.1 キー入力
- ユーザーが指定したキーを擬似入力として送信できること
- キーはGUI上のスクリーンキーボードから選択できること
- 通常のキーボードに存在しない拡張キーも選択対象に含めること
- 初期実装では誤操作リスクの低いキーを中心に提供すること
- 対応候補例:
  - `Shift`
  - `Ctrl`
  - `Alt`
  - `F13`
  - `F14`
  - `F15`
  - `F16`

#### 3.1.1.1 スクリーンキーボード選択
- キー入力方式を選択した場合、GUI内にキー選択用のスクリーンキーボードを表示できること
- スクリーンキーボード上のキーをクリックして対象キーを選択できること
- 現在選択中のキーを視覚的に強調表示すること
- 実際の物理キーボード配列に近い見た目を基本とすること
- 拡張キー群は通常キーと区別できる専用エリアまたは追加行で表示すること

#### 3.1.1.2 拡張キー対応
- `F13`から`F16`を選択可能にすること
- 将来的に`F17`以降や特殊キーを追加しやすいデータ定義構造にすること
- OSまたは入力方式の制約で送信できないキーがある場合、GUI上で無効表示または非表示にできる設計とすること

#### 3.1.2 微小なマウス移動
- マウスカーソルを視認しにくい最小限の距離だけ移動させること
- 移動後に元の位置へ戻す方式を基本とすること
- ユーザー操作への影響を最小限に抑えること

### 3.2 動作方式の選択
- GUI上で以下のいずれかを選択できること
  - キー入力
  - マウス移動
- 現在選択中の方式が画面上で明確に分かること
- キー入力方式選択時のみキー選択UIを有効表示すること

### 3.3 実行間隔の設定
- 何分ごとに擬似入力を実行するかGUIで設定できること
- 設定単位は「分」とする
- 入力値は整数で指定できること
- 最小値、最大値を定義し、不正値を防ぐこと

推奨初期仕様:
- 最小値: 1分
- 最大値: 240分
- デフォルト値: 5分

### 3.4 実行制御
- 「開始」ボタンで動作開始できること
- 「停止」ボタンで動作停止できること
- 停止中は擬似入力を行わないこと
- 動作中に設定変更した場合の挙動を明確にすること

推奨仕様:
- 動作中の設定変更は可能
- 次回実行タイミングから新設定を反映する

### 3.5 カウントダウン表示
- 稼働中は次回の擬似入力までの残り秒数をGUIに表示すること
- 表示例:
  - `次の入力まで: 142秒`
- 1秒単位で更新すること
- 停止中はカウントダウンを停止し、状態を分かるように表示すること

### 3.6 終了時刻設定
- ユーザーが指定した時刻になったらアプリの動作を終了できること
- 終了条件はGUIで有効/無効を切り替えられること
- 時刻は24時間表記で入力できること

#### 3.6.1 終了時刻の仕様
- 当日内の時刻を指定できること
- 指定時刻を過ぎている場合の扱いを明確にすること

推奨仕様:
- 現在時刻より前の時刻を指定して開始した場合は翌日の同時刻として扱う

#### 3.6.2 終了時の挙動
- 指定時刻に達したら擬似入力処理を停止すること
- 必要に応じて以下いずれかの挙動を選べる設計が望ましい
  - アプリを自動停止して待機状態に戻る
  - アプリ自体を終了する

初期仕様では以下を推奨する:
- 擬似入力を停止
- ウィンドウは閉じず、状態を「終了時刻到達により停止」と表示

### 3.7 状態表示
GUI上に最低限以下を表示すること。

- 現在状態
  - 停止中
  - 動作中
  - 終了時刻到達により停止
- 選択中の動作方式
- 次回実行までの残り秒数
- 設定中の実行間隔
- 終了時刻設定の有無

## 4. GUI要件

### 4.1 画面構成
単一ウィンドウで完結するシンプルな構成とする。

推奨レイアウト:
1. 動作方式選択エリア
2. キー選択用スクリーンキーボードエリア
3. 実行間隔設定エリア
4. 終了時刻設定エリア
5. 現在状態表示エリア
6. 開始/停止ボタンエリア

### 4.2 必須UI部品
- ラジオボタンまたはコンボボックス
  - キー入力/マウス移動の選択用
- スクリーンキーボード表示領域
  - クリックでキー選択するためのボタン群
- 選択中キー表示ラベル
  - 現在選択されたキー名の表示用
- 数値入力欄
  - 実行間隔(分)
- チェックボックス
  - 終了時刻設定の有効/無効
- 時刻入力欄
  - 終了時刻
- ラベル
  - 状態表示
  - 残り秒数表示
- ボタン
  - 開始
  - 停止
  - 終了

### 4.3 GUIライブラリ方針
軽量性を優先し、標準ライブラリ中心で構成する。

推奨:
- `tkinter`

理由:
- Python標準搭載で依存が少ない
- `.exe`化しやすい
- 軽量で常駐用途に向く

## 5. 非機能要件

### 5.1 軽量性
- 常駐時のメモリ使用量は数MB台を目標とする
- CPU使用率は待機時ほぼ0%に近いこと
- 高頻度ポーリングや不要なバックグラウンド処理を避けること

### 5.2 安定性
- 長時間稼働してもメモリリークが起きにくいこと
- GUIフリーズが起きないこと
- 例外発生時に異常終了しにくいこと

### 5.3 安全性
- ユーザーの通常操作を妨げにくいこと
- 誤入力の影響が大きいキーは初期仕様では選択対象から除外、または警告付きで扱うこと
- マウス移動は最小限とし、移動後に元の位置へ戻すこと
- スクリーンキーボード上で危険度の高いキーを選ばせない設計を優先すること

### 5.4 保守性
- 入力処理、タイマー処理、GUI処理を分離した構成とすること
- 将来的に以下の追加がしやすい設計とすること
  - キー種類の追加
  - タスクトレイ常駐
  - 起動時自動実行
  - ログ出力

## 6. 動作仕様

### 6.1 起動時
- アプリ起動時にGUIを表示する
- 初期値を読み込み、未設定でもそのまま開始できる状態にする

### 6.2 開始時
- 入力値バリデーションを実施する
- 問題なければタイマーを開始する
- 状態を「動作中」に変更する

### 6.3 動作中
- 指定秒数ごとに擬似入力を実行する
- 1秒ごとに残り秒数を更新する
- 終了時刻が有効な場合は現在時刻を監視する

### 6.4 停止時
- タイマーを停止する
- 擬似入力を中断する
- 状態表示を停止中へ変更する

### 6.5 終了時刻到達時
- 擬似入力ループを停止する
- 状態表示を更新する
- 必要であれば通知メッセージを表示する

## 7. バリデーション要件

### 7.1 実行間隔
- 数値以外を受け付けないこと
- 最小値未満、最大値超過をエラー扱いにすること
- エラー時は分かりやすいメッセージを表示すること

### 7.2 終了時刻
- `HH:MM`形式で扱えること
- 不正な時刻は開始できないこと
- 終了時刻設定が無効の場合は未入力でもよいこと

### 7.3 キー選択
- キー入力方式選択時はキー未選択のまま開始できないこと
- 非対応キーは選択不可とすること
- 拡張キー(`F13`から`F16`)が選択されている場合も通常キーと同様に保存、開始できること

## 8. 配布要件

### 8.1 EXE化
- Python未導入のWindows環境でも単体実行できること
- 単一実行ファイルまたは少数ファイル構成で配布できること

推奨手段:
- `PyInstaller`

### 8.2 配布物
最低限以下を想定する。

- `do-not-sleep.exe`
- 必要に応じて設定ファイル
- 簡易README

## 9. 技術仕様案

### 9.1 推奨構成
- GUI: `tkinter`
- タイマー: `tkinter`の`after()` または軽量なスケジューリング処理
- 入力送信:
  - Windows API利用
  - もしくは軽量な入力制御ライブラリ
- EXE化: `PyInstaller`

推奨補足:
- キー定義はGUI表示用ラベルと送信用キーコードを分離して管理する
- スクリーンキーボードは固定レイアウトのボタン配列として実装し、拡張キーを追加しやすくする

### 9.2 推奨設計モジュール
- `main.py`
  - アプリ起動
- `gui.py`
  - 画面生成とイベント処理
- `scheduler.py`
  - カウントダウンと定期実行管理
- `input_controller.py`
  - キー入力/マウス移動処理
- `config.py`
  - 初期値、設定値管理

## 11. 受け入れ条件
- GUIからキー入力またはマウス移動を選択できる
- キー入力選択時、GUI上のスクリーンキーボードから入力対象キーを選択できる
- `F13`から`F16`がGUIから選択でき、入力対象として動作する
- GUIから実行間隔を分単位で設定できる
- 動作中に残り秒数が表示される
- 指定間隔で擬似入力が実行される
- 終了時刻を設定した場合、指定時刻で停止する
- Windows上で問題なく起動する
- `.exe`形式で配布・起動できる
- 常駐時のメモリ使用量が数MB台を目標範囲に収まる設計になっている

## 12. 実装時の注意点
- 一部のセキュリティソフトや企業PC環境では擬似入力が制限される可能性がある
- フルスクリーンアプリやゲームなど一部環境では入力影響に注意が必要
- 管理者権限が必要な対象アプリには入力が届かない場合がある
- まずは安全なキー入力と最小マウス移動に限定して実装するのが望ましい
